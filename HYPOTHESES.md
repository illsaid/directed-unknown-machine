## H1 — Scenario usefulness evaluator

**Status:** killed  
**Confidence:** 0.08

**Problem statement:** People with rough tasks, notes, or ideas need a quick way to test whether a proposed tool/output would actually help in a concrete situation before building more.

**Specific user:** A solo builder deciding whether an idea is worth another hour of implementation.

**Evidence for:** Run 1 on `SCENARIOS/001-friendly.md` showed that the starter `machine.py` can read a concrete scenario and produce a structured usefulness report. Run 2 on `SCENARIOS/003-comparative.md` found that this is somewhat better than a plain checklist because it converts the scenario into a decision and next action.

**Evidence against:** Run 2 showed only a thin advantage over a disciplined checklist. Run 3 showed that evaluation alone could identify vagueness but could not turn it into useful work. Runs 5 and 6 were judged directly from their bounded output and scenario success conditions, so the usefulness-report layer added no necessary value.

**Next test:** None. Remove supporting code when that can be done without breaking the historical demo command.

**Kill criterion:** Met in Run 6: the decision-support boundary test was judged without the usefulness report.

## H2 — Labeled decision-contract shaper

**Status:** primary  
**Confidence:** 0.99

**Problem statement:** People with messy labeled decision-support notes need help turning them into one bounded decision-brief task while preserving the actual decision, supplied evidence, domain constraints, and observable success condition.

**Specific user:** A nontechnical operator trying to give an agent or contractor a bounded decision-analysis task without losing important caveats, confusing interpretation with observation, or accidentally requesting a dashboard or generic information system.

**Evidence for:** Runs 3–5 showed that the system becomes useful when it stops evaluating vague requests and instead preserves one explicit decision contract. Run 6 established a trustworthy refusal boundary for unlabeled prose; Runs 7–8 made that boundary repairable and observable without inference. Runs 9–10 transferred the unchanged four-field contract across editorial and commercial decisions. Runs 11–13 showed that unsupported structure can be rejected and repaired under the correct existing field without schema expansion. Run 14 showed that observations can live under Evidence without becoming requirements. Run 15 showed that a mixed Evidence field can preserve operator interpretation without promoting it to fact. Run 16 showed that incompatible interpretations can remain preserved and unresolved while being tested only against supplied observations. Run 17 restored inspectability by splitting accumulated reasoning obligations into explicit requirements. Runs 18–21 established auditable satisfied, violated, and unresolved constraint gates tied to supplied evidence. Run 22 on `SCENARIOS/021-mixed-independent-constraint-gates.md` showed that one satisfied and one violated constraint can coexist without being collapsed into one opaque overall verdict when each gate receives a separate evidence-backed judgment. Run 23 on `SCENARIOS/022-shared-evidence-constraint-gates.md` showed that separate gate judgments can preserve shared evidence provenance without presenting one repeated test as independent corroboration. Run 24 on `SCENARIOS/023-shared-source-partial-coverage.md` showed that one shared source can satisfy one gate while leaving another unresolved when the required measurement was never supplied. Run 25 on `SCENARIOS/024-overlapping-source-measurements.md` showed that two sources can overlap on one measurement while only one covers another gate, without letting repeated evidence spill across measurements or masquerade as broader corroboration. Run 26 on `SCENARIOS/025-conflicting-source-measurements.md` showed that incompatible values for the same measurement can remain explicit instead of being averaged or converted into false consensus. Run 27 on `SCENARIOS/026-supplied-comparability-reason.md` showed that one source can be excluded for a supplied scope mismatch without hiding either the excluded value or the reason it does not apply. Run 28 on `SCENARIOS/027-unsupported-comparability-claim.md` showed that a source cannot be excluded merely because the operator asserts it may be non-comparable; the assignment now requires a supplied observation establishing the scope mismatch and otherwise leaves the gate unresolved. Run 29 on `SCENARIOS/028-interpreted-comparability-reason.md` showed that a downstream analyst's conclusion is still not the direct observation needed to establish non-comparability when the underlying observed difference is absent. Run 30 on `SCENARIOS/029-observed-difference-uncertain-relevance.md` showed that even a direct observed scope difference cannot justify excluding a source unless supplied evidence establishes why the difference changes applicability to the target decision population. Run 31 on `SCENARIOS/030-reweighted-source-applicability.md` showed the positive boundary: supplied segment measurements, target mix, and an explicit reweighting can resolve applicability while the original unfavorable value remains visible and the adjustment method remains auditable. Run 32 on `SCENARIOS/031-opaque-adjusted-estimate.md` showed the negative boundary: a favorable adjusted number without the supplied calculation, weights, segment measurements, or other auditable method cannot resolve a gate and remains an unresolved claim alongside the original measurements. Run 33 on `SCENARIOS/032-transparent-adjustment-disputed-assumption.md` showed that reproducible arithmetic still cannot resolve a gate when a governing target-population weight is an unsupported assumption; the adjusted result must remain visible but conditional. Run 34 on `SCENARIOS/033-sensitivity-range-crosses-gate.md` showed that a supported sensitivity range crossing the decision threshold must remain an explicit conditional boundary rather than being collapsed to a midpoint, best case, or convenient point estimate. Run 35 on `SCENARIOS/034-sensitivity-range-clears-gate.md` showed the complementary boundary: a supported sensitivity range entirely below a maximum threshold can satisfy the gate without choosing a representative point or treating all uncertainty as unresolved. Run 36 on `SCENARIOS/035-sensitivity-range-violates-gate.md` showed that a supported sensitivity range entirely above a maximum threshold can establish a violated gate from the whole range, without selecting a representative value. Run 37 on `SCENARIOS/036-inclusive-threshold-touch.md` showed that a range touching a boundary can satisfy an explicitly inclusive maximum; equality must follow the supplied comparison wording rather than an assumed strict convention. Run 38 on `SCENARIOS/037-strict-threshold-touch.md` established the strict counterpart: when a supported range includes the exact maximum and the contract requires values strictly below it, equality violates the gate rather than remaining unresolved. Run 39 on `SCENARIOS/038-inclusive-minimum-touch.md` established the mirrored minimum boundary: when the lower bound of a supported range equals a threshold stated as “at least,” equality satisfies the gate rather than remaining unresolved. Run 40 on `SCENARIOS/039-strict-minimum-touch.md` established the strict-minimum counterpart: when a supported range includes the exact minimum and the contract requires values above it, equality violates the gate rather than remaining unresolved. Run 41 on `SCENARIOS/040-threshold-without-comparator.md` established the no-comparator boundary: a supplied numeric threshold whose evidence touches equality remains unresolved when the contract does not say whether equality is accepted, rather than inheriting strict or inclusive semantics. Run 42 on `SCENARIOS/041-conflicting-threshold-comparators.md` established the conflicting-comparator boundary: when the same threshold is stated as both inclusive and strict, both rules remain visible and the gate stays unresolved unless the contract supplies an explicit precedence or reconciliation rule. Run 43 on `SCENARIOS/042-conflicting-numeric-thresholds.md` established the conflicting-value boundary: when the same metric is assigned two different numeric thresholds, both values remain visible and the gate stays unresolved unless the contract supplies an explicit precedence or reconciliation rule; the system may not silently choose the stricter, looser, or later value. Run 44 on `SCENARIOS/043-equivalent-thresholds-different-units.md` established the cross-unit boundary: threshold statements in different units remain separately visible and unresolved unless the contract supplies the conversion rule or explicitly states equivalence; the system may not introduce a hidden normalization step. Run 45 on `SCENARIOS/044-explicit-cross-unit-equivalence.md` established the positive boundary: when the contract itself explicitly states that 500 milliseconds and 0.5 seconds are equivalent representations of the same threshold, the analyst may reconcile them and resolve the gate without an automatic conversion engine. Run 46 on `SCENARIOS/045-explicit-threshold-precedence.md` established the positive precedence boundary: an operator-supplied rule may identify which genuinely different threshold governs, but the displaced threshold and its separate result must remain visible rather than being silently deleted or rewritten. Run 47 on `SCENARIOS/046-grouped-requirements-preserve-precedence.md` showed that the accumulated obligations can be reorganized into seven named requirement groups while preserving the established explicit-precedence boundary and unchanged four-field contract. Run 48 on `SCENARIOS/047-consolidated-equality-semantics.md` showed that four overlapping equality clauses can be replaced by one inspectable comparison rule while preserving inclusive maximum, inclusive minimum, strict maximum, and strict minimum outcomes; the separate no-comparator boundary remains unchanged. Run 49 on `SCENARIOS/048-consolidated-threshold-conflicts.md` showed that comparator conflicts and numeric-value conflicts are instances of one preservation rule: keep every supplied threshold statement and leave the gate unresolved unless the contract supplies precedence or reconciliation. The explicit-precedence and cross-unit boundaries remain separate. Run 50 on `SCENARIOS/049-consolidated-evidence-provenance.md` showed that source identification and overlap disclosure are one provenance invariant: map each source to the measurements it actually supplies, disclose duplicated coverage, and prevent overlap from spilling into unrelated gates or masquerading as independent corroboration. Run 51 on `SCENARIOS/050-consolidated-adjustment-auditability.md` showed that preserving adjustment context, rejecting opaque methods, and conditioning transparent methods on supported assumptions are one auditability invariant: retain the original and adjusted values, method, target population, and assumptions, and do not let an unsupported assumption resolve a gate. Run 52 on `SCENARIOS/051-consolidated-source-exclusion.md` showed that justified and unjustified source exclusion share one two-part evidence invariant: preserve the excluded value and require supplied evidence of both the observed mismatch and why it changes applicability to the target population. A directly observed difference without relevance evidence remains insufficient. Run 53 on `SCENARIOS/052-consolidated-sensitivity-range.md` showed that threshold-crossing and wholly same-side sensitivity ranges are governed by one full-range invariant: preserve conditional outcomes when the range crosses the boundary, resolve the gate from the whole range when it does not, and never choose a convenient representative point. Run 54 on `SCENARIOS/053-consolidated-evidence-status.md` showed that facts, interpretations, assumptions, and gaps can be governed by one evidence-status invariant while conflicting interpretations remain governed by one observation-only comparison rule; neither interpretation becomes a measured cause, and the missing distinguishing evidence remains explicit. Run 55 on `SCENARIOS/054-consolidated-gate-judgments.md` showed that separate gate status, evidence binding, anti-collapse behavior, and recommendation blocking are one invariant: every constraint receives its own evidence-backed status, and any violated or unresolved gate remains visible and blocks a supported recommendation. Run 56 on `SCENARIOS/055-consolidated-provenance-disagreement.md` showed that source-to-measurement mapping, overlap handling, and conflicting-value preservation are one provenance invariant: preserve every source-value link, prevent shared coverage from becoming unrelated or extra evidence, and keep same-measurement disagreement unresolved unless supplied applicability evidence resolves it. Run 57 on `SCENARIOS/056-consolidated-decision-governance.md` showed that selecting one recommendation and applying the supplied success-or-reversal rule are one decision-governance invariant: require one action and show how the operator's fallback governs it, so a violated gate still produces the supplied delay recommendation rather than an ungoverned analyst preference. Run 58 on `SCENARIOS/057-consolidated-evidence-interpretation.md` showed that evidence-status preservation and observation-only treatment of conflicting interpretations are one decision-evidence invariant: preserve observations, interpretations, assumptions, and gaps as distinct statuses; do not promote claims to facts; leave unsupported conflict unresolved; and name the evidence that would distinguish it. Run 59 on `SCENARIOS/058-consolidated-applicability-transforms.md` showed that source exclusion and value adjustment are branches of one applicability-transformation invariant: preserve every original value, require supplied support specific to the transformation used, and never let an exclusion rationale substitute for an adjustment method or vice versa.

**Evidence against:** Run 4 showed the transformation does not apply to coordination problems. Run 6 showed that unlabeled prose is outside the current use case. Unsupported fields still require the operator to choose the correct destination. The executable does not classify evidence sentence by sentence or detect semantic conflict automatically; it constrains the downstream analyst, so trust still depends on the operator being able to inspect the supplied fields and fixed reasoning obligations.

**Next test:** Inspect whether the two Threshold semantics requirements can be consolidated without letting a numeric threshold with no comparator inherit strict or inclusive meaning.

**Kill criterion:** Kill if two labeled decision-support scenarios still lose the actual decision, supplied evidence, constraints, or success condition, or if preserving the boundary requires automatic semantic classification of the notes.

## H3 — Failure-mode explainer for small tools

**Status:** weakened  
**Confidence:** 0.09

**Problem statement:** Small tools often fail because nobody tests them against hostile, comparative, and transfer scenarios; users need a lightweight way to surface brittleness before relying on them.

**Specific user:** A builder with a tiny CLI or script who wants to know why it will break before sharing it.

**Evidence for:** The scenario taxonomy includes hostile, comparative, and transfer tests from the start. Run 4 exposed a concrete category error: the fixed task shaper changed a coordination problem into a decision-support problem.

**Evidence against:** The current repo has no real external tool under test except itself. Runs 5 through 59 produced their useful result by shaping, rejecting, repairing, preserving, bounding, clarifying, consistency-checking, evidence-binding, provenance-preserving, applicability-auditing, sensitivity-bounding, threshold-resolving, or structurally simplifying decision notes, not by offering a general failure explanation.

**Next test:** Do not lead implementation. Reassess only if failure analysis contributes a specific correction the decision-contract shaper could not derive directly.

**Kill criterion:** Kill if the system cannot produce failure insights more specific than obvious checklist items.
